## 一、什么是 JVM？

```
JVM = Java Virtual Machine（Java 虚拟机）

作用：
  把 Java 字节码（.class）翻译成机器码执行
  屏蔽操作系统差异，实现"一次编译，到处运行"

Java 运行流程：
  .java 源码
    ↓ javac 编译
  .class 字节码
    ↓ JVM 解释/编译执行
  机器码（操作系统执行）
```

## 二、JVM 的组织架构

```
┌─────────────────────────────────────┐
│           类加载子系统               │  加载 .class 文件
├─────────────────────────────────────┤
│              运行时数据区            │
│  ┌────┬────┬────┬────────┬───────┐  │
│  │方法区│堆  │栈  │本地方法栈│程序计数器│  │
│  └────┴────┴────┴────────┴───────┘  │
├─────────────────────────────────────┤
│           执行引擎                   │  解释器 + JIT编译器
├─────────────────────────────────────┤
│        本地方法接口（JNI）           │  调用C/C++
└─────────────────────────────────────┘
```

### 一个操作系统可以有多个 Java 程序

```
每个 Java 进程对应一个 JVM 实例
各自有独立的内存空间
互不影响
```

## 三、JVM 的内存区域 ⭐

### 五大区域

```
① 程序计数器（PC Register）
  记录当前线程执行到哪一行字节码
  线程私有，每个线程一个
  唯一不会 OOM 的区域

② 虚拟机栈（JVM Stack）
  存方法调用的栈帧
  线程私有
  每调用一个方法 → 压入一个栈帧
  方法执行完 → 弹出栈帧
  递归太深 → StackOverflowError

③ 本地方法栈（Native Method Stack）
  和虚拟机栈类似
  专门为 native 方法（C/C++代码）服务

④ 堆（Heap）
  最大的内存区域
  线程共享
  存放对象实例
  GC 主要管理这里

⑤ 方法区（Method Area）
  线程共享
  存类信息、常量、静态变量、JIT编译后的代码
  JDK8 之后叫元空间（Metaspace）
```

### 线程私有 vs 共享

```
线程私有：程序计数器、虚拟机栈、本地方法栈
线程共享：堆、方法区
```

## 四、JDK 1.6、1.7、1.8 内存变化

```
JDK 1.6：
  方法区在 JVM 内存里（永久代 PermGen）
  字符串常量池在方法区

JDK 1.7：
  字符串常量池移到堆里
  运行时常量池也移到堆里
  方法区还在（永久代）

JDK 1.8：
  永久代彻底移除
  改用元空间（Metaspace）
  元空间在本地内存（堆外），不受JVM堆大小限制
  字符串常量池还在堆里
```

---

## 五、为什么用元空间替代永久代？

```
永久代的问题：
  大小固定，容易 OOM（java.lang.OutOfMemoryError: PermGen space）
  GC 不好管理

元空间的优点：
  使用本地内存（操作系统内存），默认没有大小限制
  不容易 OOM
  类的元数据存储更灵活

注意：元空间也可以设置上限
  -XX:MaxMetaspaceSize=256m
```

## 六、对象创建的过程 ⭐

```
new Object() 发生了什么？

① 类加载检查
  检查这个类有没有被加载过
  没有 → 先加载类

② 分配内存
  在堆里为对象分配一块内存
  两种方式：
    指针碰撞：堆内存整齐，移动指针分配（Serial、ParNew）
    空闲列表：堆内存不整齐，从列表找空闲块（CMS）

③ 初始化零值
  把内存空间初始化为零值
  int=0, boolean=false, 引用=null

④ 设置对象头
  存储类信息、哈希码、GC年龄、锁状态等

⑤ 执行 <init>
  执行构造方法，初始化字段
```

---

## 七、堆内存如何分配？

### 两种分配方式

```
① 指针碰撞（Bump the Pointer）
  堆内存整齐，用过的在左，空闲的在右
  指针向右移动对象大小的距离

  [已用][已用][已用] | [空闲][空闲]
                    ↑
                   指针

② 空闲列表（Free List）
  堆内存不整齐（有碎片）
  维护一个空闲内存列表
  找到合适大小的块分配
```

### 多线程并发分配的问题

```
同时分配内存，指针被多个线程同时修改

解决：TLAB（Thread Local Allocation Buffer）
  每个线程预先分配一小块内存
  线程在自己的 TLAB 里分配，不需要加锁
  TLAB 用完了再申请新的
```

---

## 八、new 对象时，堆会发生抢占吗？

```
会！多线程同时 new 对象，都要分配内存

两种解决方案：
① 同步锁（CAS）
  用 CAS 保证指针移动的原子性
  失败了重试

② TLAB（推荐）
  每个线程有自己的内存区域
  分配时不需要和其他线程竞争
  效率更高
```

## 九、对象的内存布局 ⭐

```
Java 对象在堆里的结构：

┌─────────────┐
│   对象头     │  Mark Word（8字节）+ 类型指针（4或8字节）
├─────────────┤
│   实例数据   │  字段值（int、引用等）
├─────────────┤
│   对齐填充   │  保证对象大小是8的倍数
└─────────────┘
```

### Mark Word 存什么？

```
无锁状态：  hashCode（31位）+ GC年龄（4位）+ 偏向锁标志
偏向锁：    线程ID + epoch + GC年龄
轻量级锁：  指向栈中锁记录的指针
重量级锁：  指向 monitor 的指针
GC标记：    标记对象是否被GC
```

## 十、JVM 怎么访问对象？

### 两种方式

```
① 句柄访问
  堆里有一个句柄池
  reference → 句柄 → 对象实例数据

  优点：对象移动时只改句柄，reference不用改
  缺点：多一次寻址，慢

② 直接指针（HotSpot 使用）
  reference 直接指向对象

  优点：快，少一次寻址
  缺点：对象移动时，reference要更新
```

---

## 十一、对象有哪几种引用？

```
① 强引用（Strong Reference）
  Object obj = new Object();
  只要有强引用，GC 永远不回收
  内存不够 → OOM，也不回收

② 软引用（Soft Reference）
  SoftReference<Object> ref = new SoftReference<>(new Object());
  内存够用 → 不回收
  内存不够了 → GC 回收
  适合：缓存

③ 弱引用（Weak Reference）
  WeakReference<Object> ref = new WeakReference<>(new Object());
  只要 GC 运行，就会回收（不管内存够不够）
  适合：ThreadLocalMap 的 key

④ 虚引用（Phantom Reference）
  PhantomReference<Object> ref = new PhantomReference<>(obj, queue);
  完全没有影响对象的生命周期
  GC 回收时会收到通知
  适合：管理堆外内存
```

## 十二、Java 堆的内存分区

```
堆分为两大区域：

┌─────────────────────────────────────┐
│              新生代（Young Gen）      │  占堆的1/3
│  ┌──────────┬────────┬────────────┐ │
│  │   Eden   │  S0    │     S1     │ │
│  │  8/10    │  1/10  │    1/10    │ │
│  └──────────┴────────┴────────────┘ │
├─────────────────────────────────────┤
│              老年代（Old Gen）        │  占堆的2/3
└─────────────────────────────────────┘

Eden：新对象先在这里分配
S0/S1（Survivor）：Minor GC 后存活的对象
老年代：长期存活的对象
```

---

## 十三、新生代的区域划分

```
Eden : S0 : S1 = 8 : 1 : 1（默认比例）

新对象 → Eden
Minor GC 后存活 → S0 或 S1（两个 Survivor 交替使用）
每次 Minor GC，Eden 和当前 Survivor 的存活对象复制到另一个 Survivor
年龄达到阈值（默认15）→ 晋升老年代
```

### 为什么需要两个 Survivor？

```
只有一个 Survivor 的问题：
  存活对象和新对象都在同一块，产生碎片

两个 Survivor：
  每次 Minor GC，把存活对象从一个 Survivor 复制到另一个
  保证 Survivor 始终是干净的，没有碎片
```

---

## 十四、对象什么时候会进入老年代？

```
① 年龄达到阈值
  每次 Minor GC 后还活着，年龄 +1
  默认年龄 15 → 晋升老年代
  -XX:MaxTenuringThreshold=15 可以调整

② 大对象直接进老年代
  超过 -XX:PretenureSizeThreshold 大小的对象
  直接分配到老年代，避免频繁复制

③ 动态年龄判断
  Survivor 中相同年龄的对象总大小超过 Survivor 的一半
  大于等于该年龄的对象直接晋升老年代

④ 空间分配担保
  Minor GC 前，检查老年代剩余空间
  小于新生代所有对象总大小 → 触发 Full GC
```

## 十五、STW 了解吗？

```
STW = Stop The World

GC 时需要暂停所有用户线程
等 GC 完成后再继续

为什么要 STW？
  GC 过程中对象地址可能改变
  如果用户线程还在运行，可能读到错误的引用

STW 的影响：
  用户请求无法响应
  延迟增加，影响用户体验

优化方向：
  减少 STW 时间
  G1、ZGC 的目标就是把 STW 控制在几毫秒内
```

## 总结

| 区域 | 线程 | 存什么 | 异常 |
|---|---|---|---|
| 程序计数器 | 私有 | 当前行号 | 无 |
| 虚拟机栈 | 私有 | 方法调用栈帧 | StackOverflowError |
| 本地方法栈 | 私有 | native方法 | StackOverflowError |
| 堆 | 共享 | 对象实例 | OutOfMemoryError |
| 方法区/元空间 | 共享 | 类信息、常量 | OutOfMemoryError |

### 对象生命周期

```
new对象 → Eden分配
Minor GC存活 → Survivor（年龄+1）
年龄15 → 老年代
老年代满 → Full GC
Full GC后还不够 → OOM
```
